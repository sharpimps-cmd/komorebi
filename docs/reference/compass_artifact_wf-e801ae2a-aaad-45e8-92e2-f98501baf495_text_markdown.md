# Technical Dossier: SP803E Wi-Fi Music LED Pixel Controller

## TL;DR
- The SP803E is a **genuine mainline WLED** controller built on a **classic dual-core ESP32** (manufactured by Shenzhen Sperll Optoelectronic Technology, FCC ID 2ATV8SP803E / family grant 2ATV8SP80XE; sold by BTF-LIGHTING). It ships with real open-source WLED preinstalled — NOT a proprietary BanlanX/SPxxxE binary firmware.
- The single best path for custom software is to **use WLED's documented APIs** (JSON over HTTP, WebSocket, MQTT, UDP realtime, DDP, Art-Net/E1.31) against the stock firmware, or reflash mainline WLED via USB-C/OTA. **No reverse engineering of a binary protocol is required.**
- Key hardware facts: LED data on **GPIO16**, analog mic on **GPIO36**, 3 PWM channels on **GPIO25/26/27**, button on **GPIO0**, USB-C serial auto-download, 5–24V input, **max 2048 pixels**.

## Key Findings
- "SP803E" is a model printed on the PCB and used in the FCC filing; the OEM is Shenzhen Sperll; the primary retail brand is BTF-LIGHTING (Amazon ASIN B0FB38FDCS, ~$22).
- Sperll's own FCC declaration (dated 2022-01-17, signed by engineer Derek Xie, FCC ID 2ATV8SP80XE) states that models **SP801E through SP809E share the identical PCB, structure and internals — "Only models name is different."** So SP803E is one label in a hardware family that is entirely distinct from the older BanlanX/Sperll SP1xxE binary-protocol controllers.
- Despite the "SPxxxE" naming, the SP803E is unrelated in firmware/protocol to the closed-firmware controllers SP108E, SP107E, SP110E, SP105E, SP61xE, SP630E (controlled by apps like "LED Shop"/"LED Chord"). The SP803E runs mainline WLED and is controlled by the standard WLED app or any browser.
- The board is a classic ESP32 (not C3/S2/S3), confirmed by the GPIO map and the onboard analog mic, which WLED only supports on classic ESP32. Correct build target: `esp32dev`.

## Details

### 1. Device identification
- **What it is / OEM:** The SP803E "Wi-Fi Music LED Controller" is manufactured/certified by **Shenzhen Sperll Optoelectronic Technology Co., Ltd.** under **FCC ID 2ATV8SP803E** (family grant **2ATV8SP80XE**). It is retailed primarily by **BTF-LIGHTING** (BTF-LIGHTING product pages; Amazon ASIN B0FB38FDCS at ~$22; also sold under KEHIPI on Newegg, and on eBay/AliExpress/Alibaba). "SP803E" appears on the PCB silkscreen and in the FCC filing — it is a real device model, not merely a reseller name.
- **SoC/module:** A **classic dual-core ESP32** (ESP32-D0WD core, almost certainly an ESP32-WROOM-32/-32E module — exact silkscreen suffix unconfirmed because the FCC internal-photo exhibits could not be directly retrieved). The classic-ESP32 conclusion is essentially certain because: (a) the pin map uses GPIO36 (VP / ADC1_CH0, input-only) and GPIO26/27, which do not exist on the ESP32-C3; and (b) WLED's analog microphone support only works on the classic ESP32, and this board has an onboard analog mic on IO36. WLED documentation confirms the ADC1 requirement: *"GPIO 32–39 on ADC1 all work. Do not use any ADC2 pin (GPIO 0, 2, 4, 12–15, 25–27) — ADC2 conflicts with the WiFi radio and with I2S sampling."*
- **How SP803E relates to the SPxxxE line:** The "SPxxxE" style is a loose naming convention used by several Shenzhen vendors. The older WiFi/BT models (SP108E, SP107E, SP110E, SP105E, SP601E/602E/611E, SP630E, SP530E, SP617E) run **proprietary closed firmware** with binary socket/BLE protocols and BanlanX apps ("LED Shop," "LED Chord," "LED Hue," etc.). The **SP803E is different**: per the FCC-filed manual it is *"a high-performance LED controller based on the ESP32 module and deeply integrated with the open-source WLED project,"* runs WLED, and is controlled by the WLED app or a web browser. Critically, the Sperll FCC family grant 2ATV8SP80XE groups **SP801E–SP809E** as one identical PCB — this is the WLED-based hardware family, separate from the older 1xxE binary controllers. (An unrelated Sperll "SP801E" Art-Net product using the "BanlanX/BalanX" app also exists in listings; do not confuse marketing SKUs — rely on the FCC grant and the WLED UI to confirm what you actually have.)
- **Power / output specs (manual + listings):** Input 5V/12V/24V DC (5.5/2.1 mm barrel jack plus screw terminals); controller working current 30–150 mA; 1 SPI addressable data output on GPIO16 plus 3 PWM channels on GPIO25/26/27; PWM per-channel max 6A, total 12A; **max 2048 pixels for RGB & RGBW ICs**; dimensions 118×45×15 mm.
- **Supported LED chipsets:** SPI addressable: WS2811, WS2812/WS2812B, WS2813, WS2815, SK6812 (RGB/RGBW), WS2814 RGBW. PWM/analog: single-color, CCT (dual white), and RGB strips. (WLED itself supports clocked chipsets like APA102/SK9822, but this board exposes a single one-wire SPI data line plus PWM, so clocked chipsets would need a clock pin wired manually.)

### 2. Stock firmware and companion app
- **App(s):** The official/recommended apps are the standard **WLED** app, **WLED Native**, and **WLED+** (iOS App Store / Android APK from the WLED-App releases). There is NO SP803E-specific proprietary app; the manual instructs users to download WLED/WLED Native, connect to the "WLED-AP" access point (password `wled1234`), and browse to 4.3.2.1 to configure Wi-Fi.
- **Firmware type:** **Genuine mainline open-source WLED** (github.com/wled/WLED, formerly github.com/Aircoookie/WLED). One BTF-LIGHTING reviewer reports their unit "came with WLED v14.4" (WLED 0.14.4); another flashed the generic "_ESP32.bin" release over Wi-Fi. The standard WLED web UI, JSON API and effect set are present. It is not a disguised fork with a custom UI — it presents as stock WLED.
- **Network protocol used by the app:** Because it is stock WLED, control uses standard WLED interfaces: **HTTP JSON API** (`/json`), **WebSocket** (`/ws`), the legacy **HTTP `/win` API**, plus optional **MQTT** and **UDP realtime/sync**. Device discovery is via **mDNS/Zeroconf**. There is **no proprietary binary TCP protocol and no BLE control channel** (Wi-Fi provisioning is via the WLED captive portal, not BLE). This is the decisive difference from the SP108E-class devices, whose protocols required reverse engineering (projects like hamishcoleman/led_sp108e, kylezimmerman/pyledshop, blacklizard/LED-Shop-SP108E are NOT applicable to the SP803E).

### 3. Flashing stock/mainline WLED
- **Reflashable?** Yes. The board has an **onboard serial auto-download circuit** on the **USB-C port** (manual warns to avoid power-only USB-C cables). Flash with the WLED web installer (install.wled.me), esptool.py, or OTA. No secure-boot/encryption lockout is reported; users confirm custom builds and OTA updates work. The KEY button is GPIO0 (boot/flash); the auto-download circuit handles reset/boot over USB-C.
- **GPIO pin map (from the FCC-filed manual):**
  - LED data (SPI/addressable): **GPIO16**
  - Analog microphone: **GPIO36** (VP / ADC1_CH0, input-only, "R/O")
  - PWM channels: **GPIO25, GPIO26, GPIO27**
  - Button (KEY): **GPIO0** (short press = on/off; 1 s = change color; 10 s = reset)
  - Status LED: green indicator, steady when powered (exact GPIO not documented)
  - No IR receiver, no relay, and no exposed data-input pin (a reviewer notes adding IR/PIR requires soldering to the ESP32).
- **Flashing specifics:** UART is exposed via USB-C (onboard USB-UART + auto-reset), so manual TX/RX/GND/3V3 wiring is normally unnecessary; standard WLED flashing at 115200/460800 baud. Bricking risk is low — USB-C serial recovery is always available.
- **Build/environment target:** **`esp32dev`** (classic dual-core ESP32). For music-reactive use, an **audioreactive-enabled classic-ESP32 build** is needed (WLED 0.14+/0.15+ includes AudioReactive; select "Generic Analog" mic on GPIO36). Because BTF/Sperll already ship WLED configured for this board (LED on 16, mic on 36, PWM on 25/26/27), you can keep the vendor image and adjust settings in the UI instead of compiling.

### 4. Native/stock control protocol
Because the SP803E runs stock WLED, there is no separate proprietary protocol to reverse-engineer. "Native" control = WLED's own protocols (Section 5). Discovery is mDNS/Zeroconf; connection is HTTP on port 80, WebSocket on `/ws`, UDP sync/realtime on 21324, DDP on 4048, E1.31 on 5568, Art-Net on 6454. There is no pairing/authentication by default (optional OTA lock and optional MQTT credentials only). The SP108E/SP107E-style binary protocols (documented in pyledshop/led_sp108e) do NOT apply to this device.

### 5. WLED control APIs (the actual control surface for this device)
- **JSON API (`/json`, `/json/state`, `/json/info`, `/json/eff`, `/json/pal`, combined `/json/si`):** GET returns `{state, info, effects, palettes}`. POST partial `state` objects to control. Core state keys: `on` (bool or `"t"` toggle), `bri` (0–255), `transition`, `ps` (preset), `pl` (playlist), `nl` (nightlight), `udpn` (sync send/recv), and `seg[]`. Each segment: `start`,`stop`,`len`,`col` (up to 3 RGB(W) arrays), `fx` (effect id), `sx` (speed), `ix` (intensity), `pal` (palette id), `sel`,`rev`,`on`,`bri`,`i` (individual LED control). Examples: `{"on":true,"bri":255}`; `{"seg":[{"col":[[0,255,200]]}]}`; `{"on":"t","v":true}` toggles and returns full state (v0.13+). Since v0.14, `/json/fxdata` gives per-effect metadata (`<params>;<colors>;<palette>;<flags>;<defaults>`). Individual/range LED setting via `seg.i` arrays (hex is more efficient than RGB arrays; set brightness before setting individual LEDs; batch ≤256 colors per call, sequentially — never in parallel).
- **HTTP simple API (`/win`):** GET query-string interface, e.g. `/win&A=255` (brightness), `/win&T=1` (on), `/win&FX=0&R=255` (effect + color). Supports relative values with `~` (`PL=~` next preset, `A=~10` +10, `A=~-` wrap-decrement). Parameters include A (brightness), R/G/B/W and R2/G2/B2/W2 (colors), FX, SX, IX, FP (palette), PL (preset/playlist). Returns an XML status document. WLED recommends the JSON API for new integrations; `/win` is not deprecated.
- **UDP realtime (port 21324 by default):** Byte 0 selects protocol; Byte 1 = timeout seconds (use 1–2, or 255 = stay until other input). **WARLS** (protocol 1): index+RGB per LED (delta updates). **DRGB** (2): sequential RGB, all LEDs, max ~490 LEDs/packet. **DRGBW** (3): sequential RGBW. **DNRGB** (4): 2-byte start index + RGB, allows >490 LEDs across multiple packets. Web UI is disabled while realtime is active; HTTP/Alexa/button still work.
- **DDP (Distributed Display Protocol, UDP port 4048):** Modern realtime protocol; enable "DDP" in Sync settings and reboot. 10- or 14-byte header (WLED ignores optional timecode); data type 1/RGB888; PUSH flag on last packet of frame; max ~1440 data bytes/packet. Recommended for high-FPS streaming (used by Hyperion, xLights, LedFx). Supported since WLED v0.11.0.
- **Art-Net / E1.31 (sACN):** E1.31 default UDP port **5568** (multicast 239.255.0.x); Art-Net port **6454** (enable in Sync settings + reboot; supported since v0.10.0). 170 RGB LEDs (510 channels) per universe; up to 9 adjacent universes. DMX modes: Effect (control WLED params), Multi RGB / Multi RGBW (per-pixel; use for xLights/LedFx), single RGB/RGBW. Note: WLED 0.14.0-b1 changed the Effect-mode channel mapping (breaking change). For ~40 fps keep to ≤3 universes.
- **WebSocket (`/ws`, enabled by default since 0.10.2):** Send any JSON state update; server pushes `{state,info}` (equivalent to `/json/si`) to all clients on change and on connect. Send `{"v":true}` for full state; `{"lv":true}` for a live LED stream ("Peek"). Max 4 concurrent clients; only 1 live-stream client at a time.
- **MQTT:** Subscribes to `[deviceTopic]` (brightness 0–255 or "ON"/"OFF"/"T"), `[deviceTopic]/col` (hex `#RRGGBB`/`#WWRRGGBB` or 32-bit dec), `[deviceTopic]/api` (HTTP or JSON API payloads); plus the three `[groupTopic]` equivalents. Publishes `[deviceTopic]/g` (brightness), `/c` (color), `/v` (XML status), `/status` (LWT online/offline). Default device topic `wled/<macaddr>`; broker port 1883 (no TLS — local brokers only). HA MQTT autodiscovery was removed after 0.8.6 in favor of native HA integration.
- **Fork-specific extensions:** Since the SP803E runs mainline (not MoonModules), no fork-only API extensions are expected; flashing MoonModules WLED-MM would add extra 2D/sound features.

### 6. Music-reactive / sound input specifics
- **Hardware:** The SP803E has an **onboard analog electret microphone** feeding **GPIO36 (VP / ADC1_CH0)**. It is an analog-ADC mic (not an I2S digital mic like INMP441, and not phone-side analysis). The manual labels this "Analogmic."
- **WLED AudioReactive:** Per WLED docs, *"audio reactivity became an official usermod in WLED 0.14.0 and has been included in every official release since 0.15.0."* For this board, set microphone type to **"Generic Analog"** and GPIO to **36** in Config → Usermods → AudioReactive, then **hard-reset (RST/power cycle)** — WLED cannot reconfigure the I2S/ADC path on the fly. Analog audio only works on the classic ESP32 (which this is), so the board's wiring matches WLED's expected analog sound-reactive configuration out of the box. Caveats: ADC analog audio is noisy and susceptible to power-supply noise; per WLED docs *"analog microphones and analog buttons are mutually exclusive — WLED can use an analog microphone or analog buttons, but not both at the same time"*; and there is an open WLED issue (#4345) where the analog ADC source can read "quiet" after enabling on some builds — validate with a current audioreactive build. Audio sync can also be received over UDP multicast: WLED's sending device *"multicasts audio data to UDP multicast address 239.0.0.1, default port 11988."*

### 7. Practical integration paths
- **Home Assistant:** First-class native WLED integration (auto-discovered via mDNS/Zeroconf; requires WLED 0.14+). Uses the JSON API + WebSocket for push updates (falls back to 10 s polling if WS unavailable). Each segment becomes a light entity. Limitations: usermods (incl. AudioReactive mic toggle) are not controllable; only the primary segment color is exposed; use presets for secondary/tertiary colors and effects. No cloud/account needed.
- **ESPHome:** ESPHome does not run on this device as shipped (it runs WLED), but ESPHome devices can *send* to it via DDP/E1.31, and WLED coexists with ESPHome in HA. You could alternatively flash ESPHome to the ESP32 (neopixelbus on GPIO16), losing the WLED feature set. ESPHome's own native API (TCP, protobuf, port 6053) is unrelated to WLED control.
- **HomeKit / Google / Alexa:** WLED has built-in Alexa emulation (Philips Hue-style) for on/off/brightness/color; Google Home and HomeKit are reachable via Home Assistant bridges. Third-party iOS/watchOS apps (e.g., "ESPHome-WLED Device Manager") also control it over the JSON/WebSocket API.
- **OTA firmware update:** WLED's own OTA (enable via OTA passphrase, default `wledota`; uncheck "OTA locked," then upload a release .bin in Security settings) — standard WLED ArduinoOTA/HTTP OTA, **not** a proprietary format. You can push your own compiled WLED (or other ESP32) firmware over the air, or recover via USB-C.
- **Safety/compatibility notes:** (a) Match power-supply voltage to the strip (5/12/24 V); the controller passes input voltage straight to VCC. (b) The 3.3 V GPIO16 data line drives 5 V strips directly — for long runs/reliability use short data leads, a series resistor near the LEDs, and consider level-shifting; the manual advertises data-port surge protection and notes *"high-power drive output, utilizes high-power MOSFETs for the PWM circuit and external current amplification chips on the SPI ports"* (i.e., buffered SPI output), but an explicit 5 V level-shifter part is not documented. (c) 2048-pixel ceiling; large strips need power injection and fusing. (d) No data input for IR/PIR without soldering. (e) Reverse-polarity, backfeed, and data-port protection are advertised. Bricking risk is low due to USB-C serial recovery.

## Recommendations
1. **Start with the stock firmware + WLED JSON API.** For Claude Code, the fastest, most robust path is to target the existing WLED install: discover the device (mDNS `_wled._tcp` or its IP), GET `/json/si` to read capabilities, and POST to `/json/state` to control. Reference existing libraries (ShiftLimits/wled-client for JS/TS, paulwrath1223/wled-json-api-library for Rust, or the documented JSON schema) rather than writing a byte-level protocol.
2. **Use WebSocket (`/ws`) for live/bidirectional control** and state push; use `/json` for one-shot commands and `/win` only for quick tests.
3. **For high-FPS pixel streaming** (video/ambient/music visualizers computed off-device), use **DDP (port 4048)** or **DRGB/DNRGB UDP (port 21324)**; enable the protocol in Sync settings first.
4. **For onboard music-reactive effects,** keep or flash an AudioReactive-capable classic-ESP32 WLED build, set mic = Generic Analog on GPIO36, and hard-reset. Don't expect to toggle the mic via the HA integration — do it via the JSON/usermod API or the UI.
5. **Only reflash mainline WLED** if the vendor image is outdated or you want a custom build (e.g., MoonModules for 2D/extra sound). Use the WLED web installer over USB-C; keep the pin map (LED=16, mic=36, PWM=25/26/27, button=0). Recovery over USB-C makes risk low.
6. **Do NOT invest in reverse-engineering an SPxxxE binary protocol** — that effort applies to SP108E/SP107E-class devices, not this WLED-based board.

**Thresholds that change the recommendation:** If your specific unit ships with a *non-WLED* firmware (verify by browsing to its IP and checking for the WLED UI, or GET `/json/info` and confirm `"brand":"WLED"`), treat it like an SP108E-class device and consult the pyledshop/led_sp108e projects. If you need >2048 pixels or multiple parallel data outputs, this single-data-line board is the wrong tool.

## Caveats
- **Exact ESP32 module suffix (WROOM-32 vs -32D vs -32E vs a clone) is unconfirmed** — the FCC internal-photo exhibits for 2ATV8SP803E/2ATV8SP80XE could not be directly retrieved. The classic-ESP32 (vs C3/S2/S3) determination is nonetheless conclusive from the pin map + analog-mic requirement.
- **The 2ATV8SP80XE family grant covers SP801E–SP809E as one PCB** — feature differences between those SKUs (e.g., number of SPI channels on SP805E) are cosmetic/wiring variants of the same WLED-based design; confirm your exact port count physically.
- **Firmware version varies by production batch** — reported units shipped with WLED 0.14.x; behavior of specific JSON keys, DMX Effect-mode channel maps, and AudioReactive availability are version-dependent. Always read `/json/info` `ver`.
- **Level-shifter/capacitor presence on the SPI data output is not explicitly documented**; treat 5 V data reliability empirically.
- **The analog-ADC audio path is inherently noisy**, and WLED issue #4345 reports the analog mic reading "quiet" on some builds; validate with a current build.
- **No BLE control** — provisioning is via the WLED captive portal, not BLE (unlike some SPxxxE devices).
- Retail prices, ASINs, and app names can change; verify against current listings.
